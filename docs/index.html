Core architecture

ULLR is designed so the phone runs the experience and the TV simply displays it. This creates a low-friction living room setup where each player keeps private controls on their own device while everyone shares one big-screen output.

High-level architecture

  • Phone app: runs rendering, game logic, and controller input handling.
  • TV viewer: receives streamed output and presents the shared experience.
  • Transport: local-network streaming for low-latency screen output.
Key message: The phone runs the experience. The TV simply displays it.

Multi-device interaction model

ULLR supports multiple player phones joining the same session. Each phone acts as both a private screen and an input surface, while the TV remains the shared view of the game world.

  • Player 1 phone: private state + controller inputs.
  • Player 2 phone: private state + controller inputs.
  • TV output: synchronized shared scene for all players.

End-to-end flow

  1. Start on phone.
  2. Launch the experience.
  3. Connect to TV via same-network pairing.
  4. Stream experience to TV.
  5. Interact from phone inputs.
  6. Allow additional players to join from their phones.
  7. Keep TV focused on shared experience output.
Constraint Why it matters
Same Wi-Fi network required Enables local pairing and reliable low-latency streaming.
Phone handles rendering and processing Keeps compute on the player device and simplifies TV app footprint.
TV is display-only Keeps setup simple and focused on shared-room viewing.

Technical flow

  • Phone runtime renders game frames and captures touch, motion, and media inputs.
  • WebRTC stream carries video and audio over the local network.
  • TV viewer receives and displays output in near real time.

One-slide hero summary

Phone = engine + controller. It runs the experience and captures inputs.

TV = display. It presents the shared living room view.

Multiple phones = multiplayer. Each player joins with their own device.
